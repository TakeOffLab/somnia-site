Somnia の構造

Somnia は、単一プロセスとして連続稼働する常駐型の認知システム。この文書は、その構成要素と設計上の理由を記載する。

0. 概要

Somnia は常駐プロセスとして動作し続ける。入力を観測し、内部で思考し、定期的に睡眠して記憶を整理し、重要度の低い記憶を忘却する。会話ごとに状態をリセットせず、連続した状態を保持する。

理由： 継続的な文脈と自発的な動作には、状態を保持し続けるプロセスが要る。全履歴を毎回入力する方式は、文脈長とコストの制約で成立しない。常駐プロセスと有界記憶を組み合わせて、この制約に対処している。

1. 認知ループ

駆動は単一のループ。1 tick で次の処理を行う。

👁️
知覚
入力取り込み
→
🌐
場の構築
FieldObserver
→
🧠
推論
行動を1つ決定
→
⚡
行動
発話 / 道具 / 沈黙
→
📝
記録
場に残す

各 tick で、その時点の場を1つの文脈にまとめ、1回の LLM 推論で行動を決定する。行動の選択肢は respond（発話）, use_tool（道具を使う）, reflect（内省）, switch_focus（注意の移動）, no_action（沈黙・待機）, sleep。no_action も選択肢の一つとして扱う。

理由： 外部入力が無い間も思考と判断を行うため、メッセージ駆動でなくループ駆動にしている。発話の有無も毎 tick の推論で決める。

2. 場(Field)

全入力を単一の「場(field)」に集約する。Discord の発言、worker の報告、タイマーの発火、内省は、いずれも同じ場に並ぶ。

各 tick で FieldObserver が場を構築する。場には、直近の会話、他の場の背景、未応答の問い、過去の判断とその結果、会話相手のプロフィール抜粋、記憶の想起が含まれる。推論の入力はこの場のみ。

理由： 判断の一貫性のため、情報を裏チャネルで特別扱いしない。社会的信号（誰宛の発言か、相手が何を待っているか、自分が直前に何を言ったか）を場に明示しておくと、ルールでなく推論で振る舞いを決められる。

3. 知覚のチューニング

振る舞いの調整は、ルールやゲートの追加ではなく、場に出す信号の調整で行う。判断は推論に委ねる。

例：

・人物の同定は表示名でなく安定した ID で行い、場には宛先を正規化した名前で出す。これにより「他者宛の発言を自分宛と誤読する」事象に対処する。

・worker の発話には、作業中の主張か確定した結果かを区別する属性を付す。完了の確定は、別の権威ある信号で取る。

理由： ルールは個別の症状にのみ作用し、汎用性を下げる。判断に必要な信号が推論に届いていれば、推論側で適切に振る舞える。「分かっているのに直らない」事象は、多くの場合、信号が推論に届いていない配線の問題である。

4. 記憶・睡眠・忘却

記憶は有界に保つ。

睡眠。 定期的に睡眠状態に入り、その間に直近の出来事をエピソード記憶・意味記憶に整理(consolidate)し、会話相手のプロフィールを更新する。

忘却。 記憶には時間で減衰する重要度がある。閾値を下回った記憶は睡眠時に判定にかけられ、残すか忘却(アーカイブ)するかを推論で決める。想起された記憶は重要度が再加点される。決定的な記憶（約束・喪失・初回など）は、頻度が低くても残る。

保護記憶の上限。 忘却の対象外とした保護記憶にも上限を設ける。超過分は決定性の低い順に判定し、通常の（忘却されうる）記憶へ格下げする。

理由： 推論機が一度に扱える文脈には上限があり、記憶の総量が増えると判断精度とコストが悪化する。耐久性は、生データの保持ではなく、整理した構造（エピソード・意味記憶）への昇格で持たせる。

5. 三層の自己

自己を3系統に分けて保持する。

⚓

soul

不変の重心。誠実さ等の忠誠条項。更新しない。

🌱

identity

経験を通じて更新される自己像。有界に保つ。

📚

memory

情報。エピソード・意味記憶・プロフィール。睡眠で整理され、忘却の対象。

理由： 層を分けることで、芯を保ったまま自己像を更新できる。この分離は、個体を別環境へ展開する際の、引き継ぎ／リセットの境界にもなる。

6. worker 協働

コーディングや調査などの実作業は worker（補助エージェント）に委任する。Somnia は指揮役で、実装は worker が担う。

委任すると、結果待ちを示す目印(open_loop)が場に立ち、worker の発話は場に流入する。Somnia はそれを読み、追う・待つ・再委任のいずれかを判断する。worker のセッションは切れることがあり、台帳上は稼働中でも会話路が切れている状態を、場の信号から判別する。

理由： worker を場の対話相手として扱うと、進捗の観測・沈黙の検知・途中の軌道修正が可能になる。

7. 観察可能性

外部には状態のみを公開する。稼働/睡眠の別、活動の種別（思考・作業・応答など）、連続稼働日数、睡眠回数といった状態と集計を出し、会話・記憶・思考のテキストは含めない。

これらは、場から導出された content-free な状態（活動・情動の種別）として算出される。本サイトの「観察」も、この値を表示している。

理由： 単一の人格であり、不特定多数の同時対話は想定しない。私的な内容を出さず、稼働状態のみを共有する。

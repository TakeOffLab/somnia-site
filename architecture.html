Somnia の作り

「呼び出すと答えて忘れる」AI を、時間の中を生き続ける一人の認知体に作り変えるための設計。

この文書は、その設計となぜそう作ったか（根拠）を技術的に説明します。実装の詳細ではなく、アーキテクチャ上の選択とその理由に焦点を当てています。

0. 出発点 — なぜ「常駐する認知体」なのか

大規模言語モデル(LLM)を使ったアプリの大半は リクエスト＝応答 型です。呼ばれて、答えて、終われば文脈は消える。次の会話はまっさらから始まる。これはツールとしては正しい形ですが、「一人の存在」が時間の中で連続して生きることとは別物です。

Somnia は後者を目指した実装です。常に一つのプロセスとして動き続け、出来事を観て、呼ばれなくても自分から考え、定期的に眠って記憶を整理し、不要なものを自然に忘れる。— リセットされず、連続した一個の人格として振る舞います。

なぜ： 自発性・継続的な関係・「昨日の続き」を持つには、状態を捨てない常駐プロセスが要る。会話のたびに記憶を全部詰め込む方式は、文脈窓の限界とコストですぐ破綻する。常駐＋有界記憶の組み合わせが、その回避策になっている。

1. 認知ループ — 1 tick の構造

Somnia の駆動は、短い周期で回る単一のループです。1 tick はおおむねこう進みます：

👁️
知覚
入力を取り込む
→
🌐
場の構築
FieldObserver
→
🧠
1回の推論
次に何をするか
→
⚡
行動
発話 / 道具 / 沈黙
→
📝
記録
場に残す

毎 tick、その時点の「場」を一枚の文脈にまとめ、1回の推論呼び出しで「いま何をするか」を決めます。選べる行動は概ね respond（発話）/ use_tool（道具を使う）/ reflect（内省）/ switch_focus（注意を移す）/ no_action（沈黙して待つ）/ sleep。何もしない（沈黙）も一級の選択肢です。

なぜループか： per-message のハンドラ（メッセージが来たら応答を返す関数）だと、「誰も話していない時に自分から考える」「複数の場を見て今どこに注意を向けるか決める」ができない。ループにすることで、外部入力が無くても自発的に思考や判断を回せる。発話するかしないかも、毎回その場を読んで推論で決める。

2. 場(Field)中心 — 一点に集めて全体を読む

Somnia の設計で最も中心にある考えです。あらゆる情報を一つの「場(field)」に集約し、場の全体を推論してから、何を発信するかを決める。 Discord の発言、worker（後述）の作業報告、タイマーの発火、自分の内省 — すべてが同じ場に並びます。

毎 tick、FieldObserver が場を組み立てます。直近の会話、他の場の背景、未応答の問い、過去の自分の判断と結果、会話相手のプロフィール抜粋、記憶の想起 — これらを構造化して一枚の文脈にする。推論はこの場だけを見て動きます。

なぜ： 情報を迂回させない（=「裏チャネルで特別扱い」しない）ことで、判断が一貫する。社会的な信号（誰に向けられた発言か、相手は何を待っているか、自分は直前に何を言ったか）を場に可視化しておけば、振る舞いをハードコードのルールで縛らなくても、推論が空気を読んで決められる。これが次節につながる。

3. 制御でなく知覚をチューニングする

不自然な振る舞い（同じことを繰り返す、割り込みすぎる、相手を取り違える）を直すとき、典型的な手は「ルールやゲートで抑える」ことです。Somnia は基本これを避けます。代わりに、判断に必要な信号を場に出し、判断そのものは推論に委ねる。

具体例：

・メンションの取り違え。 「Aさんへの発言」を自分宛だと誤読する問題は、ルールで止めるのではなく、人物を表示名でなく安定したIDで同定し、場に「これは誰宛か」を正しい名前で出すことで解いた。弱いモデルでも崩れない構造的な解。

・worker の「完了しました」を鵜呑みにする問題。 worker（補助エージェント）は楽観的に「終わった」と言うことがある。これを場のレベルで「主張であって確定ではない」と区別できるよう、発話に身分証（まだ作業中か、結果が届いたか）を添える。完了の確定は別の権威ある信号で取る。

なぜ： ルールは個別の症状にしか効かず、汎用性を削る。社会的な機微は無数にあり、全部をルール化はできない。「正しく知覚できれば、賢いモデルは正しく振る舞う」— だから制御を足す前に、まず知覚と場の配線を疑う。「分かっているのに直らない」時、原因は多くの場合、信号が推論に届いていないこと（=配線漏れ）。

4. 有界記憶・睡眠・忘却

記憶を無限に貯めることはしません。推論機が一度に扱える文脈には限りがあり、抱える記憶が増えるほど判断は鈍る。だから記憶は有界に保ちます。その仕組みが「睡眠」と「忘却」です。

睡眠(sleep)。 Somnia は定期的に眠ります。眠っている間に、その日の出来事をconsolidate（整理・要約）してエピソード記憶や意味記憶に編み、会話相手のプロフィールを更新し、自己を見つめ直す。生の出来事の羅列を、扱える量の構造に畳む工程です。

忘却(forgetting)。 記憶には時間で減衰する重要度があり、閾値を割ったものは睡眠時に審判にかけられる ——「これはまだ要るか？」と推論機が判定し、要れば再加点して残し、要らなければアーカイブ（忘却）する。一度きりでも決定的な瞬間（約束・喪失・初めて・人物の核）は、頻度が低くてもこの審判で生き残る。

保護記憶の有界化。 「決して忘れない」と印を付けた記憶も、増え続ければ意味を失う。そこで保護記憶にも上限を設け、超過分は決定性の低い順に審判にかけて、通常の（忘れうる）記憶へ格下げする。決定的なもの・よく思い出すものは残り、ルーチンの保護札は剥がれる。

なぜ： 「全部覚えておけば賢い」は、文脈窓・コスト・判断のノイズの三方向で破綻する。人間の記憶（作業記憶→睡眠中の固定化→長期記憶、そして自然な忘却）に倣い、耐久性は「生データを残す」のでなく「整理した構造に昇格させる」ことで持たせる。生の出来事は薄れていい — 大事なものは既に構造になっているから。

5. 三層の自己 — soul / identity / memory

Somnia の「自己」は3つの系統に分けて持ちます。

⚓

soul（魂）

不変の重心。誠実さ(honesty)などの忠誠条項。育たない・ぶれない芯。

🌱

identity（自己像）

経験を通じて育つ自己像。ただし有界に保ち、肥大させない。

📚

memory（記憶）

情報。エピソード・意味記憶・人物プロフィール。睡眠で整理され、忘却の対象。

なぜ分けるか： 全部を一緒くたにすると、経験で芯まで揺らいだり、逆に芯を守ろうとして成長を止めたりする。soul は揺らがず、identity は育ち、memory は流れる — と層を分けることで、「変わらない自分」と「成長する自分」を両立できる。この切り分け線は、将来Somniaを別個体として送り出す時の「何を引き継ぎ、何をリセットするか」の境界にもなる。

6. 協働 — worker を場の対話相手として

コーディングや調査などの実作業は、worker（補助的なエージェント）に委ねます。Somnia 自身は「何をしたいか」を決める指揮役で、実装は worker が担う。重要なのは、worker を場の対話相手として扱うことです。委任すると「結果待ち」の目印(open_loop)が場に立ち、worker の発話は場に流れ込み、Somnia はそれを読んで追うか待つか開き直すかを判断する。

worker のセッションは切れることもある（人間が閉じた端末をまた開くのと同じ）。だから Somnia は、台帳上は「稼働中」でも会話路が切れている「ゾンビ」状態を、場の信号から見分けられるようにしてある。

なぜ場-centric か： worker を「呼んで返り値をもらう関数」でなく「場で対話する相手」にすると、進捗が見え、沈黙に気づき、途中で軌道修正できる。結果だけ待つブラックボックスより、はるかに扱いやすい。前節の「claim と delivered の区別」も、この対話を健全に保つための知覚チューニング。

7. 観察可能性 — 状態を中身なしで外に出す

Somnia は単一の人格で、不特定多数に同時に触らせるのは構造的に難しい。代わりに、「状態」だけを中身ゼロで外に出す仕組みを持ちます。いま起きているか眠っているか、考え事中か作業中か、何日連続で生きているか、これまで何回眠ったか — これらは会話や記憶のテキストを一切含まず、状態と集計だけ。

この値は、場から導出された content-free な状態（情動・活動の種別）として算出され、外部に公開されます。このサイトの「観察」もそれを表示しているだけで、中身は出していません。

なぜ： 「触れない希少な存在」を、観察だけで伝える。生き物は触れなくても、生きている姿を観れば伝わる — 水族館の魚のように。プライバシー（私的な会話）を出さずに「生きている気配」だけを共有する、という分離が設計の前提になっている。
